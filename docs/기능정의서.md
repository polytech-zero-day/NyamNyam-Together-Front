## 2. 상세 기능 정의 (Feature Definition)

### F-01 모임 목적 선택 (그룹 생성 · Must · B·공통)
- **설명**: 모임 목적을 선택해 질문 세트를 결정한다.
- **입력**: 모임 목적(친구/회식/점심)
- **출력**: 질문 템플릿 결정
- **처리 규칙**: ① 목적이 질문 세트 결정 ② MVP는 친구 모임만 완성 ③ 그 외 목적은 구조만 확장 가능
- **예외**: 친구 외 목적은 "준비중" 비활성
- **비고**: 불필요한 질문 차단 = 결정 피로 감소

### F-02 조건 입력 (그룹 생성 · Must · B)
- **설명**: 생성자가 모임 고정 조건을 입력한다.
- **입력**: 최소 인원·위치·마감
- **출력**: 그룹 생성·식별자 발급
- **처리 규칙**: ① 위치·인원·마감은 생성자 1회 고정(참여자 미노출) ② 위치=서울 주요 역 반경
  ③ 마감=자동 종료 트리거 ④ **생성자(host)=토스 로그인 필수**
- **예외**: 필수값 누락·과거 마감 시 차단

### F-03 초대 링크 발급 (그룹 생성 · Must · B)
- **설명**: 참여용 초대 링크를 발급한다.
- **입력**: (그룹 생성 완료)
- **출력**: 초대 링크
- **처리 규칙**: ① 그룹당 고유 링크 ② 링크로 통째 입장 ③ 종료 전까지 유효
- **예외**: 종료된 링크는 결과 화면 안내

### F-04 링크로 입장 (참여 · Must · B·공통)
- **설명**: 참여자가 링크로 입장한다.
- **입력**: 초대 링크
- **출력**: 세션 참여(등록)
- **처리 규칙**: ① **host(그룹 생성·종료)만 토스 식별, 참여자는 익명(무로그인)** — 링크로 입장
  ② 종료 전 입·퇴장 자유 ③ 늦게 입장 시 투표 단계만 합류
- **예외**: 종료된 그룹은 결과 화면
- **비고**: host=토스 userKey / 참여자=익명 토큰(음수 id). 로그인 약관은 콘솔 등록 약관 사용.

### F-05 술 수용도 응답 (질문 세션 · Must · B 수집·C 매핑)
- **설명**: 술자리 수용 정도를 응답한다.
- **입력**: 3지선다(①마실것 / ②안 마셔도 괜찮음 / ③술 없는 곳이 편함)
- **출력**: 장소타입 결정 제약
- **처리 규칙**: ① ③ 1명이라도 → 일반 위주 ② ②만/①+② → 양립+일반 ③ ①포함+③없음 → 전부
  ④ google types → place_type 매핑(추천엔진 ② 참조)
- **예외**: 미응답 시 진행 불가
- **비고**: 사람 배제 않는 보수적 방향

### F-06 예산 범위 입력 (질문 세션 · Must · B 수집·C 컷)
- **설명**: 1인 기준 예산 범위를 입력한다.
- **입력**: 1인 기준 예산 — **구간 다중선택**(1만 이하 / 1~2만 / 2~3만 / 3~4만 / 4만 이상, 중복선택 가능)
- **출력**: 예산 제약(min/max)
- **처리 규칙**: ① 선택 구간들의 **최저 하한=min, 최고 상한=max**로 환산.
  비연속 선택도 최저~최고를 하나의 연속 범위로 처리(중간 구간 포함) ② 범위 → priceLevel 밴드 환산
  ③ 상한(max) 주력 컷(P25 완충), 하한(min) 소프트 ④ priceLevel null이면 통과(미적용) ⑤ 부드러운 필터(0개 방지)
- **예외**: (구간 미선택 시 선호 미반영 / 진행 안내)
- **비고**: 예: "1~2만"+"2~3만" 선택 → 1만~3만 범위

### F-07 음식 카테고리 다중선택 (질문 세션 · Must · B 수집·C 매칭)
- **설명**: 선호 음식 카테고리를 복수 선택한다.
- **입력**: 한글 카테고리 N개 (한식/일식/양식/중식/분식/아시안/고기·구이/카페·브런치, 최대 3개)
- **출력**: 선호 집합(정렬용)
- **처리 규칙**: ① 거르지 않고 정렬에만 사용 ② 2표 이상만 채택(합집합 폭발 방지)
  ③ 겹칠수록 높은 매칭 점수 ④ 한글 분류 → google types 매칭
- **예외**: 미선택 시 선호 미반영

### F-08 분위기 응답 (질문 세션 · Should · C)
- **설명**: 선호 분위기 성향을 응답한다.
- **입력**: 분위기 성향 선택(편하게 먹는 / 도란도란 대화 / 왁자지껄한 / 분위기 있는)
- **출력**: 분위기 가중치
- **처리 규칙**: ① 거르지 않음(후보 풀 유지) ② 평균 성향만 산출 ③ **MVP에선 가중치 0(사실상 미사용)**
- **비고**: Atmosphere 필드(최고 티어) 미사용 → 추후 등록 데이터로 보강

### F-09 종료 트리거·스냅샷 집계 (로딩화면 · Must · B)
- **설명**: 종료 시점을 감지해 응답을 집계한다.
- **입력**: 수동 종료/마감 도달
- **출력**: 집계 결과(제약)
- **처리 규칙**: ① 수동/마감 중 먼저 ② 종료 전 미집계 ③ 스냅샷 1회 집계.
  ※ votes 원본 집계(예산 종합·다수결·표수)는 B파트
- **예외**: 최소 인원 미충족 시 보류
- **비고**: 집계 결과를 추천엔진(C) 입력으로 전달

### F-10 추천 엔진 (로딩화면 · Must ★핵심 · C)
- **설명**: 집계 제약으로 후보를 필터링하고 선택 정렬로 3~4곳 압축한다.
- **입력**: 집계 제약 + 정렬 기준(리뷰순/평점순/랜덤) + 역 좌표
- **출력**: 식당 후보 3~4곳
- **처리 규칙**: ① 후보 풀 = 구글 Nearby 라이브 결과 ∪ 등록(owner/community) 식당
  ② 필터: 술→place_type / 예산→priceLevel / 위치→역 반경 / 음식→2표 매칭(선호)
  ③ 정렬: 선택 기준 적용(추천엔진 ⑦) ④ 상위 3~4곳 압축
  ⑤ ToS: place_id만 저장, 평점·가격 등은 라이브 조회 후 폐기
- **예외**: 0개 시 완화(F-11)
- **비고**: Google Places(New) · place_id만 영구저장

### F-11 후보 0개 시 완화 + 공지 (공지화면 · Must · C)
- **설명**: 후보가 없으면 조건을 단계적으로 완화·공지한다.
- **입력**: 빈 후보
- **출력**: 완화 후보 + 공지 배너
- **처리 규칙**: ① 완화 순서: 예산(하한 먼저) → 음식 카테고리 → 위치 반경
  ② 술 제약은 끝까지 유지 ③ 완화 시 명시 공지
- **예외**: 끝까지 0개면 안내 문구
- **화면 구성**: 안내 문구 → 완화 항목 before→after 표시 → "술자리 조건은 그대로 지켰어요" → "추천 결과 보기" 버튼
- **비고**: 데모 안정성. relaxed=true 배너

### F-12 후보 카드 표시 (투표 · Must · C·전원)
- **설명**: 압축된 후보를 카드로 표시한다.
- **입력**: 후보 식당 목록
- **출력**: 후보 카드 UI
- **처리 규칙**: ① 후보별 카드 ② 이름·거리·지도·평점(rating)·리뷰수(userRatingCount) 표시 — 구글 라이브
  ③ 누락 항목은 표시 생략 ④ "Powered by Google" 출처 표기
- **비고**: AI 추천 이유 없음(v1.6 제거)

### F-13 투표하기 (투표 · Must · B·전원)
- **설명**: 후보에 👍를 눌러 그룹 선호를 모은다.
- **입력**: 후보 카드 👍 탭
- **출력**: 득표 집계·최다 하이라이트
- **처리 규칙**: ① 후보=세션당 1세트(모두 동일) ② 👍=개인 1표
  ③ 같은 화면에서 수렴, 최다 득표 자연 부상 ④ 최종 선택은 사용자 몫(압축기)
- **예외**: 늦게 입장도 👍 단계 참여 가능

### F-14 응답 진행률 표시 (로딩(투표 후) · Must · B)
- **설명**: 응답 진행 상황을 표시한다.
- **출력**: 진행률(예: 3/5명)
- **처리 규칙**: 실시간 응답자 수 / 전체 참여자 수

### F-16 정렬 기준 선택 (투표 전·정렬 · Must · C)
- **설명**: 후보 노출 직전, 정렬 방식을 1개 선택한다.
- **입력**: 리뷰순/평점순/랜덤 (3지선다)
- **출력**: 정렬 기준 확정
- **처리 규칙**: ① 1차 투표 종료 후, 후보 노출 직전
  ② 리뷰순=userRatingCount / 평점순=rating(리뷰 10개 미만 신뢰 하향) / 랜덤=시드 고정 셔플
  ③ 세션 공유 설정(개인별 아님), 후보·집계 불변 ④ 선택값을 F-10 정렬에 전달
- **예외**: 미선택 시 기본값(리뷰순)
- **비고**: 기본=리뷰순(표본 얇은 한국 식당 보정). 백엔드는 sort 파라미터 수신

### F-17 점주·시민 식당 등록 (식당 등록 · Could · C)
- **설명**: 구글에 없는 식당을 직접 등록한다.
- **입력**: source/역/이름/좌표/카테고리/가격/개업일
- **출력**: 등록 식당(first-party)
- **처리 규칙**: ① MVP는 화면 스텁 + first-party 저장(검증·심사 추후)
  ② source=owner/community로 구분 ③ 개업일(open_date)=운영연차 신호
  ④ 추후 추천 풀 합류 시 평점 없는 등록 식당은 배지 구분
- **예외**: 필수값 누락 시 차단
- **비고**: 구글 표본 얇은 한국 식당 보완 확장축

## 3. 추천 엔진 규칙 (필터 → 정렬 → 압축 · Google Places(New) · place_id만 저장)

### ① 파이프라인 개요

| 단계 | 유형 | 사용 제약 | 역할 |
|---|---|---|---|
| 입력 | — | B가 집계한 제약 + 역 좌표 · 후보 풀 = 구글 Nearby 라이브 ∪ 등록 식당 | 파이프라인 입력 |
| 1단계 | 필터형 | 술→장소타입 / 예산→priceLevel 컷 / 위치→역 반경 / 음식→2표 매칭(선호) | 후보 풀 자르기 |
| 2단계 | 정렬형 | 선택 기준 — 리뷰순(기본)/평점순/랜덤 · 분위기는 MVP 미사용 · longevity 가점 | 선택 기준 정렬 |
| 3단계 | 압축 | 상위 3~4곳 | 최종 후보 압축 |

### ② 술 수용도 → 장소타입 (google types 매핑)

| 응답 조합 | 후보 장소타입 |
|---|---|
| ① 포함 + ③ 없음 | 술필수 + 양립 + 일반 (전부) |
| ②만 / ①+② (③ 없음) | 양립 + 일반 (술필수 제외) |
| ③ 1명이라도 | 일반 위주 |

| place_type | google types |
|---|---|
| 술필수 | `bar`, `pub`, `wine_bar`, `night_club` |
| 양립 | `barbecue_restaurant`, `brewery`, `bar_and_grill` (포차·고깃집 보강 룩업) |
| 일반 | 그 외 `*_restaurant`, `cafe`, `bakery`, `coffee_shop` |

### ③ 예산 집계 규칙 (구간 다중선택 · 상한 주력)

| 규칙 | 내용 |
|---|---|
| 입력 | 구간 다중선택(1만 이하~4만 이상). 최저 하한=min, 최고 상한=max로 환산 → priceLevel(1~4) 매핑. 비연속 선택도 최저~최고 연속 범위로 처리 |
| 상한(max) | 주력 필터. 상한들 중 낮은 쪽이 강하게(P25 완충, 0개 방지) |
| 하한(min) | 소프트 — 하드 필터 아님. min 미만은 정렬 약한 후순위. 0개 시 가장 먼저 완화 |
| null 처리 | priceLevel null인 식당 → 예산 필터 미적용(통과) |
| (선택) priceRange | 한국 커버리지 확인 후 원 단위 정밀화 — 미도입 시 밴드 기준 |

### ③-1 음식 카테고리 집계 규칙

| 규칙 | 내용 |
|---|---|
| 채택 기준 | 2표 이상 받은 카테고리만 채택(합집합 폭발 방지) |
| 점수화 | 채택 카테고리와 google types가 겹칠수록 높은 매칭 점수(필터 아님, 정렬용) |

### ④ 분위기 집계 규칙

| 규칙 | 내용 |
|---|---|
| 필터 여부 | 거르지 않음(후보 풀 유지) |
| MVP 사용 | 가중치 0(사실상 미사용) — Atmosphere 필드(goodForGroups 등)는 최고 티어라 미수집 |
| 추후 | 등록 식당 데이터 등으로 보강 시 재도입 |

### ⑤ 후보 0개 시 조건 완화 순서

| 순서 | 완화 대상 | 유지 조건 |
|---|---|---|
| 1 | 예산 하한(min) 완화 → 상한 완충폭 확대 | 술 제약 끝까지 유지 |
| 2 | 음식 카테고리 | 술 제약 끝까지 유지 |
| 3 | 위치 반경(Nearby radius 확대 1회 재호출) | 술 제약 끝까지 유지 |

※ 공지: "조건에 맞는 곳이 없어 일부 조건을 완화해 추천했어요" (relaxed=true 배너)

### ⑥ 데이터·캐싱·ToS 전략 (Google Places New · place_id만 저장)

| 항목 | 내용 |
|---|---|
| 소스 | Google Places API(New). Nearby Search 1회=최대 20개(과금 1건). 풀 부족 시 타입별 다중 호출로 확장 |
| 필드마스크 | Enterprise 단일 호출: types·primaryType·location + rating·userRatingCount·priceLevel. ❌ Atmosphere(reviews 본문·editorialSummary·servesXxx·goodForGroups) 금지 |
| 저장(ToS) | place_id만 영구 저장(면제 필드). 평점·이름·가격 등 콘텐츠는 저장 금지 → 라이브 조회 후 세션 내 사용·폐기(오프라인 적재 안 함) |
| 캐시 | 역별 place_id 목록(station_places) 30일 디스커버리 주기 재사용. 최종 후보 3~4곳만 Place Details 라이브 |
| stale | 조회 시 CLOSED_PERMANENTLY 비활성 / movedPlaceId 교체. 지도 없이 노출 시 "Powered by Google" 표기 |
| 출처 구분 | places.source ∈ google / owner / community. google=place_id+가공값만, 등록=first-party 전체 |

### ⑦ 정렬 기준 선택 규칙 (1차 투표 후 · 세션 공유 · 1개 선택)

| 기준 | 정렬 키 | 처리 / 동점 규칙 |
|---|---|---|
| 리뷰순 (기본) | userRatingCount ↓ | 동점 시 rating ↓ · null/등록식당 후순위(감점 아님) |
| 평점순 | rating ↓ | userRatingCount ≥ 10 미만 신뢰 하향 · 동점 시 userRatingCount ↓ |
| 랜덤 | session.sort_seed | 시드 고정 셔플 — 새로고침해도 순서 불변 |

### ⑧ 운영연차(longevity) 규칙 [신규]

| 규칙 | 내용 |
|---|---|
| 출처 | 등록(owner) 식당의 open_date 만 사용. 구글 openingDate는 미래 개업 전용이라 미사용 |
| 작용 | 오래 운영 = 상권 생존 방증 → 약한 가점. open_date 없으면 0(감점 아님) |

### ⑨ 비용·한도 메모

| 항목 | 내용 |
|---|---|
| 무료 한도 | Enterprise SKU 월 1,000건. Nearby 1회=식당 20개 → seed 역+세션 규모는 무료 구간 |
| 절감 레버 | 데이터 가공 저장이 아니라 place_id 목록 재사용 + 최종 후보만 라이브가 실제 호출 절감 |
