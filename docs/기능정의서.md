## 상세 기능 정의 (Feature Definition)

### F-01 모임 목적 선택 (그룹 생성 / Must / 생성자)
- 설명: 모임 목적을 선택해 이후 참여자 질문 세트를 결정한다.
- 입력값: 모임 목적(친구 모임 / 직장 회식 / 점심)
- 출력/결과: 선택 목적에 맞는 질문 템플릿 결정
- 처리 규칙: ① 목적이 질문 세트를 결정 ② MVP는 '친구 모임'만 완성(술+예산+음식+분위기) ③ 그 외 목적은 구조만 확장 가능하게 준비
- 예외/유효성: MVP에서 친구 모임 외 목적은 '준비중'으로 비활성 처리
- 비고: 불필요한 질문 차단 = 결정 피로 감소

### F-02 조건 입력 (그룹 생성 / Must / 생성자)
- 설명: 생성자가 모임의 고정 조건을 입력해 그룹을 생성한다.
- 입력값: 최소 인원, 위치, 마감 시간
- 출력/결과: 그룹 생성, 그룹 식별자 발급
- 처리 규칙: ① 위치·인원·마감은 생성자가 1회 고정(참여자 질문에 미노출) ② 위치는 서울 주요 역 반경 기준 ③ 마감 시간은 자동 종료 트리거로 사용
- 예외/유효성: 필수값 누락 시 생성 불가, 마감 시간이 현재 이전이면 입력 차단

### F-03 초대 링크 발급 (그룹 생성 / Must / 생성자)
- 설명: 생성된 그룹의 참여용 초대 링크를 발급한다.
- 출력/결과: 공유 가능한 초대 링크
- 처리 규칙: ① 그룹당 고유 링크 발급 ② 링크로 모임 전체가 통째로 입장 ③ 종료 전까지 유효
- 예외/유효성: 종료·마감된 그룹 링크 진입 시 결과 화면으로 안내

### F-04 링크로 입장 (참여 / Must / 참여자)
- 설명: 참여자가 초대 링크로 질문 세션에 입장한다.
- 입력값: 초대 링크
- 출력/결과: 세션 참여(참여자 등록)
- 처리 규칙: ① 앱인토스 사용자 식별 활용(자체 로그인 없음) ② 종료 전 입·퇴장 자유 ③ 늦게 입장 시 마지막 투표 단계만 합류
- 예외/유효성: 종료·마감된 그룹은 입장 대신 결과 화면 노출

### F-05 술 수용도 응답 (질문 세션 / Must / 참여자)
- 설명: 참여자가 술자리 수용 정도를 응답한다.
- 입력값: 3지선다 (①마실 것 / ②안 마시지만 술자리 괜찮음 / ③술자리 불편)
- 출력/결과: 참여자 술 제약 (장소타입 결정에 사용)
- 처리 규칙: ① 응답을 모아 장소타입 결정 ② ③이 1명이라도 있으면 일반 위주 ③ ②만/①+②(③없음) = 양립+일반 ④ ①포함+③없음 = 전부 (상세: 추천엔진 규칙)
- 예외/유효성: 미응답 시 다음 단계 진행 불가
- 비고: 사람 배제하지 않는 보수적 방향. 양립 예시 업종: 이자카야·포차·고깃집·호프·곱창

### F-06 예산 범위 입력 (질문 세션 / Must / 참여자)
- 설명: 참여자가 1인 기준 예산 범위를 입력한다.
- 입력값: 1인 기준 최소~최대 금액 범위 (예: 2만~3만)
- 출력/결과: 참여자 예산 제약
- 처리 규칙: ① 참여자 상한 중 낮은 쪽이 강하게 작용 ② 완전 최솟값은 위험 → 하위 25% 또는 최솟값~중앙값 사이로 완충 ③ 부드러운 필터(후보 0개 방지)
- 예외/유효성: 최소 > 최대 입력 시 안내 후 재입력

### F-07 음식 카테고리 다중선택 (질문 세션 / Must / 참여자)
- 설명: 참여자가 선호 음식 카테고리를 복수 선택한다.
- 입력값: 음식 카테고리 N개 (한식 / 고기 / 일식 / 중식 / 양식 / 아시안 / 분식 / 간편식(샌드위치) / 치킨,안주)
- 출력/결과: 참여자 선호 집합
- 처리 규칙: ① 거르지 않고 정렬에만 사용 ② **2표 이상 받은 카테고리만 채택**(합집합 폭발 방지) ③ 채택 카테고리와 겹칠수록 높은 매칭 점수 ④ 선호일 뿐 필터가 아님
- 예외/유효성: 미선택 시 선호 미반영(후보 풀 유지)

### F-08 분위기 응답 (질문 세션 / Should / 참여자)
- 설명: 참여자가 선호 분위기 성향을 응답한다.
- 입력값: 분위기 성향 선택
- 출력/결과: 분위기 선호 가중치
- 처리 규칙: ① 거르지 않음(후보 풀 유지) ② 참여자 평균 성향만 산출 ③ 동점 후보 간 순위 조정 가중치로만 사용(약한 가중치)
- 비고: 음식점 결정의 후순위 제약. "차분한 게 좋아요" → 룸/조용한 음식점, "상관 없어요" → 모든 곳

### F-09 종료 트리거 감지·스냅샷 집계 (로딩화면 / Must / 생성자 수동·시스템 자동)
- 설명: 종료 시점을 감지해 응답을 스냅샷으로 집계한다.
- 입력값: 수동 종료 또는 마감 시간 도달
- 출력/결과: 장소타입·예산·선호 집계 결과
- 처리 규칙: ① 생성자 수동 종료 또는 마감 도달 중 먼저 오는 것 ② 종료 전에는 집계하지 않음(입·퇴장 자유) ③ 종료 시점 응답만 스냅샷으로 집계
- 예외/유효성: 최소 인원 미충족 시 종료/집계 보류 및 안내
- 화면 느낌: "다른 사람의 선택을 기다리고 있어요" 로딩

### F-10 매핑+플레이스 → 후보 압축 (로딩화면 / Must / 시스템) ★핵심 엔진
- 설명: 집계 결과로 2단 파이프라인을 돌려 후보 식당을 압축한다.
- 입력값: 집계 결과(장소타입·예산·위치·선호)
- 출력/결과: 식당 후보 3~4곳
- 처리 규칙:
  1. 1단계 필터형: 술→장소타입, 예산→`avg_price_min` 기준 컷(null이면 미적용), 위치→역 반경
  2. 2단계 선호형: 음식→2표 이상 매칭, 분위기→약한 가중치, `source_rating`(다이닝코드·식신)→null이면 카카오 accuracy 순서 유지
  3. Claude API 웹서치로 최종 3~4곳 선별 + `ai_reason` 생성
  4. 전 과정 카카오 목록 기반(할루시네이션 차단)
- 예외/유효성: 후보 0개 시 조건 완화 기능(F-11) 호출
- 화면 느낌: "여러분의 선택으로 음식점을 고르고 있어요" 로딩

### F-11 후보 0개 시 완화 + 공지 (공지화면 / Must / 시스템)
- 설명: 후보가 없을 때 조건을 단계적으로 완화하고 사용자에게 알린다.
- 입력값: 빈 후보 결과
- 출력/결과: 완화된 후보 목록 + 완화 공지 배너
- 처리 규칙: ① 완화 순서: 예산 완충폭 확대 → 음식 카테고리 → 위치 반경 ② 술 제약은 끝까지 유지 ③ 완화 시 명시적 공지 노출
- 예외/유효성: 끝까지 완화해도 0개면 안내 문구로 마무리
- 화면 느낌: "앗! 원하시는 음식점이 없어 범위를 조정하고 있어요" 공지

### F-12 후보 카드 표시 (투표 / Must / 전원)
- 설명: 압축된 후보를 카드 리스트로 표시한다.
- 입력값: 후보 식당 목록
- 출력/결과: 후보 카드 UI
- 처리 규칙: ① 후보별 카드 노출 ② 이름·거리·지도·AI추천이유 표시. **평점은 카카오 미제공 → `source_rating` 표시, 없으면 생략**
- 예외/유효성: 정보 누락 항목은 표시 생략

### F-13 투표하기 (👍) (투표 / Must / 전원)
- 설명: 후보에 👍를 눌러 그룹 선호를 모은다.
- 입력값: 후보 카드 👍 탭
- 출력/결과: 득표 집계, 최다 득표 하이라이트
- 처리 규칙: ① 후보 = 전체 집계 기반 1세트(모두 동일하게 봄) ② 👍 = 개인별 1표, 앱인토스 user ID 기준 중복 방지 ③ 같은 화면에서 수렴, 최다 득표 후보 자연 부상 ④ **최종 한 곳 선택은 사용자 몫(압축기, 확정기 아님)**
- 예외/유효성: 늦게 입장한 참여자도 👍 단계부터 참여 가능

### F-14 응답 진행률 표시 (투표 후 확인 로딩 / Must / 전원)
- 설명: 참여자 응답 진행 상황을 표시한다.
- 출력/결과: 진행률 표시 (예: '3/5명 응답')
- 처리 규칙: 실시간 응답자 수 / 전체 참여자 수 표시

### F-15 AI 추천 이유 한 줄 (투표 / Should / 전원)
- 설명: 선정된 후보에 추천 이유를 한 줄로 덧붙인다.
- 입력값: 후보 식당
- 출력/결과: 추천 이유 설명 텍스트
- 처리 규칙: ① 카카오 목록 기반으로 Claude 웹서치(다이닝코드·식신 우선) 실제 정보 확인 ② 최종 3~4곳 선별 후 `ai_reason` 한 줄 생성 ③ `confidence:low`(확인 불가) 제외 ④ AI 응답 실패 시 설명 생략(후보는 그대로 노출)

## 추천 엔진 규칙 (2단 파이프라인 — 전부 코드)

### ① 파이프라인 개요

| 단계 | 유형 | 사용 제약 | 역할 |
|---|---|---|---|
| 1단계 | 필터형 | 술 → 장소타입 / 예산 → `avg_price_min` 기준 컷 / 위치 → 역 반경 | 후보 풀 자르기 |
| 2단계 | 선호형 | 음식 → 2표 이상 합집합 매칭 / 분위기 → 약한 가중치 / `source_rating`(다이닝코드·식신) → null이면 카카오 accuracy 순서 유지 | 남은 후보 정렬 |
| 3단계 | 압축 | 상위 3~4곳 | 최종 후보 압축 |

### ② 술 수용도 매핑 규칙

| 응답 조합 | 후보 장소타입 |
|---|---|
| ① 포함 + ③ 없음 | 술필수 + 양립 + 일반 (전부) |
| ②만 / ①+② (③ 없음) | 양립 + 일반 (술필수 제외) |
| ③ 1명이라도 있음 | 일반 위주 |

| 장소타입 | 예시 업종 |
|---|---|
| 술필수 | 칵테일바, 와인바, LP바, 펍 |
| 양립 | 퓨전포차, 고깃집, 이자카야, 호프 |

### ③ 예산 집계 규칙

| 규칙 | 내용 |
|---|---|
| 입력 | 참여자별 최소~최대 범위 입력 |
| 작용 방향 | 참여자 상한 중 낮은 쪽이 강하게 작용(가장 빠듯한 사람도 감당 가능) |
| 완충 | 완전 최솟값은 위험 → 하위 25% 또는 최솟값~중앙값 사이로 완충(후보 0개 방지) |

### ③-1 음식 카테고리 집계 규칙

| 규칙 | 내용 |
|---|---|
| 채택 기준 | 2표 이상 받은 카테고리만 채택(합집합 폭발 방지) |
| 점수화 | 채택 카테고리와 겹치는 가게일수록 높은 매칭 점수(필터 아님, 정렬용) |

### ④ 분위기 집계 규칙

| 규칙 | 내용 |
|---|---|

| 사용 | 동점 후보 간 순위 조정 가중치로만 사용(후순위 제약) |

### ⑥ 데이터 수집 전략 (카테고리별 키워드 합집합)

| 방식 | 내용 |
|---|---|
| 사전 배치 키워드 | 이자카야·포차·고깃집·호프·곱창·한식·중식·일식·양식·분식·해산물·치킨·피자·스테이크·라멘 + FD6 전체 |
| 수집 방식 | 키워드별 45개 검색 → `kakao_id` 기준 dedup → 역당 150~250개 저장 |
| search_keywords | 수집 시 사용한 키워드 누적 기록 (text[] 배열) |
| 웹서치 보완 | Claude API로 다이닝코드·식신 참조 → `price_level`·`mood`·`source_rating` 저장 (확인된 것만, null 허용) |
| 실시간 세션 | DB SQL 1차 필터 → Claude API 2차 최종 선별 (카카오 API 호출 없음) |
