## 1. 진실 공급원 (SOURCE OF TRUTH) — 충돌 시 우선순위

문서가 여러 개라 내용이 충돌하는 부분이 있다. 충돌 시 **아래 순서로 우선**한다.

1. **`docs/기능정의서.md` (기능정의서 v1.7 + 추천엔진 규칙)** ← 최우선. 사람이 직접 작성한 핵심.
2. `docs/PRD.md`
3. `docs/기능명세서.md` ← AI가 자동 생성한 문서라 일부 부정확. 아래 "정정 사항"을 반드시 적용.

> ⚠️ v1.4 → v1.7에서 데이터 소스·인증·기능 구성이 크게 바뀌었다.
> **카카오 API / Claude 웹서치 / 오프라인 적재 / AI 추천 이유(F-15)는 모두 폐기됐다.**
> 옛 버전 문서나 옛 코드가 보이면 v1.7 규칙으로 덮어쓴다.

### 1.1 기능명세서.md 정정 사항 (이걸 따르지 말 것 → 기능정의서가 맞음)

- **술 수용도는 2지선다(O/X)가 아니라 3지선다다.**
  ① 술 마실 거예요 ② 안 마시지만 술자리는 괜찮아요 ③ 술자리 자체가 불편해요
  (추천 엔진의 술→장소타입 매핑이 이 3지선다에 의존하므로 절대 O/X로 만들지 말 것)
- **예산은 "구간 다중선택"으로 입력받아 min/max 범위로 환산한다. (UI 확정)**
  구간: 1만원 이하 / 1~2만 / 2~3만 / 3~4만 / 4만 이상. **중복선택 가능**(체크박스).
  → 선택한 구간들의 **가장 낮은 하한 = min, 가장 높은 상한 = max**로 환산한다.
  → 비연속 선택(예: "1만 이하" + "3~4만"만 체크, 가운데 비움)도 **최저~최고를 하나의 연속 범위로 본다(A안).**
    즉 위 예시는 "1만 이하 ~ 4만"으로 처리(중간 구간 식당도 포함). 구멍을 그대로 두지 않는다.
  → 환산된 min/max를 추천 엔진 3.3(상한 주력·하한 소프트)에 그대로 넘긴다.
- **음식은 "선호 다중선택"만 구현한다.** 비선호 입력·자동완성은 MVP 범위 밖(만들지 말 것).
  **2표 이상 받은 카테고리만 채택**한다(합집합 폭발 방지). 필터가 아니라 정렬용.
  한글 카테고리 → google types로 매칭한다.
- **로그인 약관 내용은 콘솔에 등록된 약관을 사용한다. (F-04 / 로그인 바텀시트)**
  로그인 화면 바텀시트의 약관 항목·문구는 앱에 하드코딩하지 말고, **앱인토스 콘솔에 등록된 약관**을
  표시한다. (약관 텍스트를 코드에 직접 박지 말 것)
- **인증은 host만 토스 식별, 참여자는 익명이다. (v1.7 핵심 변경)**
  그룹 생성·종료(host)만 토스 로그인(userKey) 필수. **참여자는 무로그인으로 링크 입장**(익명 토큰, 음수 id).
  → 참여자에게 토스 로그인을 강제하지 말 것. 이름/닉네임도 따로 입력받지 않는다.
- **분위기(F-08)는 MVP에서 가중치 0(사실상 미사용)이다.** Atmosphere 필드(goodForGroups 등)는
  구글 최고 티어라 미수집. 화면은 두되 추천 점수엔 반영하지 않는다. 우선순위 Should.
- **실시간 득표는 단순 갱신으로 충분하다.** 웹소켓 등 복잡한 실시간 동기화는 필수 아님.
- **위치는 주소 "검색"이 아니라 "역 목록에서 선택"이다.** (권역 선택 → 역 선택 2단)
  역 목록은 백엔드 API로 받아온다(미완성 시 목업 데이터로 대체).
- **식당 데이터는 구글 Places API(New) 단일 소스다. (v1.7 핵심 변경)**
  카카오 API·Claude 웹서치(다이닝코드·식신)는 **폐기**. 평점(`rating`)·리뷰수(`userRatingCount`)·
  가격(`priceLevel`)은 구글이 직접 제공한다. **`place_id`만 영구 저장**하고 나머지 콘텐츠는
  라이브 조회 후 폐기(ToS 준수, 아래 3.5 참고).
- **정렬 기준 선택(F-16) 화면이 신설됐다(Must).** 후보 노출 직전 리뷰순(기본)/평점순/랜덤 중 택1.
- **AI 추천 이유(F-15)는 폐기됐다.** 후보 카드에 AI 한 줄 이유를 넣지 말 것(외부 LLM 의존 제거).

## 2. 절대 규칙 (항상 지킬 것)

- **추천(필터·정렬·선정)은 100% 코드로 처리한다.** AI가 식당을 임의로 생성하지 않는다.
  (할루시네이션 금지 / 재현 가능성 확보). 식당 데이터는 **구글 Places API(New)**에서만 가져오며,
  **목록 밖 식당을 생성하지 않는다.** (외부 LLM 추천 이유 생성도 폐기 — F-15)
- **UI는 TDS(Toss Design System) Mobile 컴포넌트만 사용한다.** 커스텀 스타일·임의 컴포넌트 금지.
  TDS Mobile 문서(`@docs/skills/tds-mobile.md`)를 참조해 정확한 컴포넌트와 props를 쓴다.
- **모든 화면 최상단은 TDS Navigation을 둔다.**
- **라이트 모드, 세로(모바일) 기준.** primaryColor = `#FF5F00`. 앱 아이콘은 콘솔에 업로드한 로고 사용.
- **인증: host만 토스 식별, 참여자는 익명(무로그인).** 그룹 생성·종료만 토스 userKey 필수.
  참여자에게 로그인을 강제하지 않는다.
- **구글 Places ToS 준수: `place_id`만 영구 저장한다.** 평점·이름·가격 등 콘텐츠는 저장 금지 →
  세션 내 라이브 조회 후 폐기(오프라인 적재 금지). 지도 없이 노출 시 "Powered by Google" 표기.
- **MVP는 '친구 모임' 목적 하나만 완성한다.** 그 외 목적(직장 회식/점심)은 구조만 확장 가능하게
  두고 비활성("준비중").
- **제외 기능은 만들지 않는다**: 참여자 자체 로그인, AI 식당 임의 생성, **AI 추천 이유(폐기됨)**,
  카카오 API·외부 웹서치, 식당 콘텐츠 오프라인 적재, 실제 예약, 결제, 더치페이 정산,
  종료 푸시 알림, 음식 비선호 입력, 그룹 히스토리/즐겨찾기, 실시간 채팅.
- **B파트 / C파트 소유 경계를 지킨다.** (아래 9장 참고) B=호스트·세션·집계·투표 백엔드,
  C=추천엔진·구글 Places·정렬·식당 카드·등록. 한쪽 영역을 임의로 침범하지 않는다.
- AI가 만든 코드는 사람이 검수한 뒤에만 다음 단계로 넘어간다. (AI 실수는 팀 책임)

## 3. 핵심 로직 — 추천 엔진 (절대 임의로 바꾸지 말 것)

### 3.1 파이프라인 (필터 → 정렬 → 압축, 전부 코드, v1.7 기준)

- **입력**: B가 집계한 제약 + 역 좌표. 후보 풀 = **구글 Nearby 라이브 결과 ∪ 등록(owner/community) 식당**
- **1단계 (필터형 — 후보 풀 자르기)**
  - 술 수용도 → 장소 타입 결정 (술필수 / 양립 / 일반) → google types 매핑
  - 예산 → `priceLevel`(1~4) 밴드 컷. 상한(max) 주력, 하한(min) 소프트. null이면 통과(미적용)
  - 위치 → 선택한 역 반경 (Nearby radius)
  - 음식 → 2표 이상 받은 카테고리 매칭(선호, 필터 아님)
- **2단계 (정렬형 — 남은 후보 정렬)**
  - **F-16에서 사용자가 고른 정렬 기준 적용**: 리뷰순(기본)/평점순/랜덤 (아래 3.6)
  - 분위기 → **MVP 가중치 0(미사용)**
  - 운영연차(longevity) → 등록 식당 open_date 기반 약한 가점 (아래 3.7)
- **3단계 (압축)** → 상위 **3~4곳** 선별
  - ToS: place_id만 저장, 평점·가격 등은 라이브 조회 후 폐기. AI 추천 이유 생성 없음(F-15 폐기).

### 3.2 술 수용도 매핑 규칙 (→ google types)

| 참여자 응답 조합 | 후보 장소 타입 |
|---|---|
| ①(마심) 포함 + ③(불편) 없음 | 술필수 + 양립 + 일반 **전부** |
| ②만 / ①+② (③ 없음) | **양립 + 일반** (술필수 제외) |
| ③(불편) 1명이라도 있음 | **일반 위주** |

- 술필수 (google types): `bar`, `pub`, `wine_bar`, `night_club`
- 양립 (google types): `barbecue_restaurant`, `brewery`, `bar_and_grill` (포차·고깃집 보강 룩업)
- 일반 (google types): 그 외 `*_restaurant`, `cafe`, `bakery`, `coffee_shop`

### 3.3 예산 집계 (구간 다중선택 → priceLevel 밴드, 상한 주력)

- **입력 UI**: 구간 다중선택(1만 이하 / 1~2만 / 2~3만 / 3~4만 / 4만 이상, 체크박스).
  선택 구간의 **최저 하한=min, 최고 상한=max**로 환산. 비연속 선택도 최저~최고 연속 범위로 처리(A안).
- 각자 환산된 `최소~최대` 범위 → 허용 `priceLevel`(1~4) 구간으로 매핑.
- **상한(max)**: 주력 필터. 참여자 상한 중 **낮은 쪽이 강하게** 작용(가장 빠듯한 사람 기준), P25 완충.
- **하한(min)**: 소프트(하드 필터 아님). min 미만은 정렬 약한 후순위. 0개 시 **가장 먼저 완화**.
- `priceLevel`이 null인 식당 → 예산 필터 미적용(통과).
- (선택) priceRange: 한국 커버리지 확인 후 원 단위 정밀화. 미도입 시 밴드 기준 유지.

### 3.4 후보 0개 시 조건 완화 (F-11)

- 완화 순서: **예산 하한(min) 완화 → 상한 완충폭 확대 → 음식 카테고리 → 위치 반경(Nearby radius 확대 1회 재호출)**
  (술 제약은 끝까지 유지)
- 완화 시 사용자에게 **명시적 공지**: "조건에 맞는 곳이 없어 일부 조건을 완화해 추천했어요" (relaxed=true 배너)

### 3.5 데이터 출처 / ToS (v1.7 — 구글 Places New 단일 소스)

- **식당 데이터는 구글 Places API(New) 단일 소스다.** 카카오 API·Claude 웹서치는 폐기.
  - 목록: Nearby Search 1회 = 최대 20개(과금 1건). 풀 부족 시 타입별 다중 호출로 확장.
  - 필드: `types`·`primaryType`·`location` + `rating`·`userRatingCount`·`priceLevel`.
    ❌ Atmosphere 필드(reviews 본문·editorialSummary·servesXxx·goodForGroups) 호출 금지(최고 티어).
- **저장(ToS 핵심)**: **`place_id`만 영구 저장**(면제 필드). 평점·이름·가격 등 콘텐츠는 저장 금지 →
  라이브 조회 후 세션 내 사용·폐기. **오프라인 적재 금지.**
- **캐시**: 역별 place_id 목록(`station_places`) 30일 디스커버리 주기 재사용.
  최종 후보 3~4곳만 Place Details 라이브 조회.
- **stale 처리**: 조회 시 `CLOSED_PERMANENTLY` 비활성 / `movedPlaceId` 교체.
  지도 없이 노출 시 **"Powered by Google"** 표기.
- **출처 구분**: `places.source ∈ google / owner / community`.
  google = place_id+가공값만, 등록(owner/community) = first-party 전체 저장.
- **API 키는 클라이언트에 박지 말고 서버/함수에 둔다.**

### 3.6 정렬 기준 선택 (F-16 — 1차 투표 후 · 세션 공유 · 1개 선택)

| 기준 | 정렬 키 | 처리 / 동점 규칙 |
|---|---|---|
| **리뷰순 (기본)** | `userRatingCount` ↓ | 동점 시 `rating` ↓ · null/등록식당 후순위(감점 아님) |
| 평점순 | `rating` ↓ | `userRatingCount` ≥ 10 미만 신뢰 하향 · 동점 시 `userRatingCount` ↓ |
| 랜덤 | `session.sort_seed` | 시드 고정 셔플 — 새로고침해도 순서 불변 |

- 미선택 시 **기본값=리뷰순**(표본 얇은 한국 식당 보정).
- **세션 공유 설정(개인별 아님)**, 후보·집계는 불변. 선택값을 F-10 정렬에 전달. 백엔드는 sort 파라미터 수신.

### 3.7 운영연차(longevity) 가점 [v1.6 신규]

- 출처: **등록(owner) 식당의 `open_date`만** 사용. 구글 `openingDate`는 미래 개업 전용이라 미사용.
- 작용: 오래 운영 = 상권 생존 방증 → **약한 가점**. `open_date` 없으면 0(감점 아님).

## 4. 핵심 사용자 플로우

```
[생성자] 모임 생성(목적·최소인원·역·마감) → 초대 링크 공유     ※ host = 토스 로그인 필수
[참여자] 링크 입장(무로그인·익명) → 제약 질문 응답(술·예산·음식·분위기, 3스텝 이내)
[종료]   생성자 수동 종료 OR 마감시간 도달 (먼저 오는 것)
[시스템] 종료 시점 스냅샷 → 집계 → (정렬 기준 선택 F-16) → 후보 3~4곳 압축
[전원]   후보 👍 투표 → 최다 득표 자연 부상
```

- 참여자 기준 입장→응답→투표 **3스텝 이내**.
- 종료 전엔 집계하지 않음 → 입·퇴장 자유. 늦게 온 사람은 투표 단계부터 합류.
- **투표 = 추천된 후보 3~4곳 중에서 고르는 것**(취향 투표가 아님). 추천이 먼저, 투표가 나중.
- **정렬 기준 선택(F-16)은 1차 투표 종료 후, 후보 노출 직전**에 1회(세션 공유).
- 👍는 **개인별 1표, 중복 방지**. host는 토스 userKey, 참여자는 익명 토큰 기준으로 식별.
- 최종 한 곳 선택은 **사용자 몫**(시스템은 최다 득표 하이라이트만, 확정하지 않음).

## 6. 데이터 / 외부 연동

- **식당 데이터**: 구글 Places API(New) — `types`·`primaryType`·`location`·`rating`·`userRatingCount`·`priceLevel`.
  Nearby Search 1회 = 최대 20개. Atmosphere 필드(reviews·editorialSummary·goodForGroups 등) 호출 금지.
- **ToS 저장 규칙**: `place_id`만 영구 저장. 평점·이름·가격 등 콘텐츠는 라이브 조회 후 폐기(오프라인 적재 금지).
  지도 없이 노출 시 "Powered by Google" 표기. **API 키는 클라이언트에 박지 말고 서버/함수에 둔다.**
- **장소 타입 분류**: 사람이 다듬은 룩업 테이블(google types → place_type 매핑). 코드로 적용, 재현 가능.
- **역 목록**: 백엔드 API로 받아온다. (서울 권역 + 인천 + 광명 기준, 권역→역 2단 선택)
- **인증**: host = 토스 userKey(그룹 생성·종료만). 참여자 = 익명 토큰(음수 id), 무로그인 링크 입장.
- **백엔드/DB**: 프론트와 분리하지 않고 같은 레포에 서버리스(Supabase) 방식으로 구성한다.
  저장 대상: 그룹 / 참여자(익명 토큰) / 제약 응답 / 후보(place_id) / 투표 / 정렬기준(sort_seed) /
  역별 place_id 캐시(station_places) / 등록 식당(owner·community first-party).
  (스키마 확정은 백엔드 담당)

## 8. 작업 방식 (하네스) — 화면설계서 → 코드 변환

이 프로젝트의 기본 작업 흐름은 **① 피그마 화면설계서 → ② TDS Mobile React 코드 → ③ 샌드박스 검증 →
④ 사람 검수**의 반복이다. 아래 순서를 따른다.

1. **화면설계서를 진실로 삼는다.** 각 화면(F-01~F-17)은 피그마 화면설계서가 시각적 기준이다.
   캡처는 `docs/screens/`에, 피그마 원본은 링크로 참조한다. 시안에 없는 요소를 임의로 추가하지 않는다.
2. **화면설계서 → 코드 변환 방법** (둘 중 가능한 것 사용):
   - (권장) **Figma MCP**로 해당 프레임을 읽어 TDS Mobile 컴포넌트로 변환한다.
     레이아웃·간격·색상은 시안을 따르되, **실제 UI 요소는 반드시 TDS Mobile 컴포넌트로 치환**한다
     (피그마가 그린 사각형을 그대로 div로 만들지 말 것).
   - (대안) MCP 미연결 시 `docs/screens/`의 캡처를 보고 동일 기준으로 TDS 컴포넌트로 구현한다.
3. **TDS 우선 원칙.** 버튼·시트·인풋·리스트·내비게이션 등은 `@toss/tds-mobile`의 컴포넌트를 먼저 찾아 쓴다.
   커스텀 스타일은 TDS로 표현 불가능할 때만, 최소 범위로. (2장 절대 규칙)
4. **변환 후 자기설명.** 코드를 짠 뒤 **어떤 TDS 컴포넌트를 왜 썼는지 한 줄씩** 설명한다(사람 검수용).
   시안과 다르게 구현한 부분이 있으면 그 이유도 명시한다.
5. **자가수정 루프.** 타입 체크(`npx tsc --noEmit`)·빌드·샌드박스 실행에서 나온 에러 메시지를 그대로 받아
   수정 → 다시 실행. 통과할 때까지 반복하되, 추측으로 로직을 바꾸지 않는다.
6. **한 화면씩.** 여러 화면을 한꺼번에 변환하지 말고 화면 단위로 변환→검증→검수한 뒤 다음으로 넘어간다.
7. **로직은 규칙대로.** 추천 엔진·예산(priceLevel)·술 매핑(google types)·완화·정렬(F-16)·운영연차 로직은
   3장의 규칙을 **정확히** 코드로 옮긴다. 임의 변경 금지. (카카오·웹서치·AI 추천 이유는 폐기됐으니 쓰지 말 것)
8. **B/C 경계 준수.** 9장의 파트 구분을 지킨다. 내 파트가 아닌 영역은 인터페이스(입출력)만 맞추고 침범하지 않는다.

## 9. 파트 소유 경계 (B / C) — v1.7 신규

문서가 B파트/C파트 소유를 명시한다. 작업 시 이 경계를 지킨다.

| 파트 | 소유 범위 | 해당 기능 |
|---|---|---|
| **B** | 호스트 인증·세션·집계·투표 백엔드 | F-02·F-03·F-04(입장)·F-09·F-13·F-14, votes 원본 집계 |
| **C** | 추천엔진·구글 Places·정렬·식당 카드·등록 | F-05~F-08(매핑)·F-10·F-11·F-12·F-16·F-17 |
| 공통 | 양쪽이 함께 보는 화면/식별 | F-01·F-04(공통), 제약 응답 수집 UI |

- **인터페이스만 맞추고 침범 금지.** 예: C는 B가 넘겨준 "집계 제약"을 입력으로 받아 추천만 한다.
  B의 votes 집계 로직이나 host 인증 흐름을 C가 임의로 건드리지 않는다.
- 한 사람이 양쪽을 다 구현하더라도, 모듈/폴더를 B·C로 나눠 경계를 코드에서도 유지한다.
