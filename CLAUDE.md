# 냠냠투게더 (nyam-nyam-together)

> 앱인토스(Apps in Toss) 미니앱 · WebView(Vite) · TDS Mobile 기반
> 이 파일은 Claude Code가 프로젝트를 시작할 때 자동으로 읽는 컨텍스트다. 항상 이 규칙을 따른다.

## 0. 무엇을 만드나 (한 줄)

모임 조건(위치·인원)과 참여자 제약(술·예산·음식·분위기)을 모아, 갈 만한 식당을
**3~4곳으로 압축**해주는 그룹 식당 결정 도구. 최종 한 곳은 사용자가 고른다.
→ **"확정기가 아니라 압축기"**

자세한 내용은 `docs/`의 PRD·기능정의서·기능명세서 참고.

## 1. 진실 공급원 (SOURCE OF TRUTH) — 충돌 시 우선순위

문서가 여러 개라 내용이 충돌하는 부분이 있다. 충돌 시 **아래 순서로 우선**한다.

1. **`docs/기능정의서.md` (기능정의서 v1.4 + 추천엔진 규칙)** ← 최우선. 사람이 직접 작성한 핵심.
2. `docs/PRD.md`
3. `docs/기능명세서.md` ← AI가 자동 생성한 문서라 일부 부정확. 아래 "정정 사항"을 반드시 적용.

### 1.1 기능명세서.md 정정 사항 (이걸 따르지 말 것 → 기능정의서가 맞음)

- **술 수용도는 2지선다(O/X)가 아니라 3지선다다.**
  ① 술 마실 거예요 ② 안 마시지만 술자리는 괜찮아요 ③ 술자리 자체가 불편해요
  (추천 엔진의 술→장소타입 매핑이 이 3지선다에 의존하므로 절대 O/X로 만들지 말 것)
- **음식은 "선호 다중선택"만 구현한다.** 비선호 입력·자동완성은 MVP 범위 밖(만들지 말 것).
  **2표 이상 받은 카테고리만 채택**한다(합집합 폭발 방지). 필터가 아니라 정렬용.
- **참여자는 이름/닉네임을 따로 입력받지 않는다.** 앱인토스 사용자 식별을 활용한다(자체 로그인 없음).
- **분위기는 성향 선택**이며 약한 가중치로만 쓴다(거르지 않음). 우선순위 Should.
- **실시간 득표는 단순 갱신으로 충분하다.** 웹소켓 등 복잡한 실시간 동기화는 필수 아님.
- **위치는 주소 "검색"이 아니라 "역 목록에서 선택"이다.** (권역 선택 → 역 선택 2단)
  역 목록은 백엔드 API로 받아온다(미완성 시 목업 데이터로 대체).
- **평점·리뷰는 카카오 API가 제공하지 않는다.** Claude API 웹서치(다이닝코드·식신)로
  `source_rating`을 보완하며, 없으면(null) 표시를 생략한다. (아래 3.5 참고)

## 2. 절대 규칙 (항상 지킬 것)

- **추천(필터·정렬·선정)은 100% 코드로 처리한다.** AI가 식당을 임의로 생성하지 않는다.
  (할루시네이션 금지 / 재현 가능성 확보). 단, 평점·분위기·추천이유는 Claude API 웹서치로 보완하되
  **반드시 카카오 플레이스 목록 안에서만** 선별한다(목록 밖 식당 생성 금지).
- **UI는 TDS(Toss Design System) Mobile 컴포넌트만 사용한다.** 커스텀 스타일·임의 컴포넌트 금지.
  TDS Mobile 문서(`@docs/skills/tds-mobile.md`)를 참조해 정확한 컴포넌트와 props를 쓴다.
- **모든 화면 최상단은 TDS Navigation을 둔다.**
- **라이트 모드, 세로(모바일) 기준.** primaryColor = `#FF5F00`. 앱 아이콘은 콘솔에 업로드한 로고 사용.
- **MVP는 '친구 모임' 목적 하나만 완성한다.** 그 외 목적(직장 회식/점심)은 구조만 확장 가능하게
  두고 비활성("준비중").
- **제외 기능은 만들지 않는다**: 자체 로그인, AI 식당 임의 생성, 실제 예약, 결제, 더치페이 정산,
  종료 푸시 알림, 음식 비선호 입력, 그룹 히스토리/즐겨찾기, 실시간 채팅.
- AI가 만든 코드는 사람이 검수한 뒤에만 다음 단계로 넘어간다. (AI 실수는 팀 책임)

## 3. 핵심 로직 — 추천 엔진 (절대 임의로 바꾸지 말 것)

### 3.1 2단 파이프라인 (전부 코드, v1.4 기준)

- **1단계 (필터형 — 후보 풀 자르기)**
  - 술 수용도 → 장소 타입 결정 (술필수 / 양립 / 일반)
  - 예산 → `avg_price_min` 기준 보수적 상한 컷 (null이면 미적용, 완충 적용)
  - 위치 → 선택한 역 반경
- **2단계 (선호형 — 남은 후보 정렬)**
  - 음식 카테고리 → **2표 이상** 받은 카테고리만 채택, 겹칠수록 높은 매칭 점수
  - 분위기 → **약한 가중치** (거르지 않음, 동점 순위 조정용)
  - `source_rating`(다이닝코드·식신) → null이면 **카카오 accuracy 순서 유지**
- **3단계 (압축)** → Claude API 웹서치로 최종 **3~4곳** 선별 + `ai_reason` 생성

### 3.2 술 수용도 매핑 규칙

| 참여자 응답 조합 | 후보 장소 타입 |
|---|---|
| ①(마심) 포함 + ③(불편) 없음 | 술필수 + 양립 + 일반 **전부** |
| ②만 / ①+② (③ 없음) | **양립 + 일반** (술필수 제외) |
| ③(불편) 1명이라도 있음 | **일반 위주** |

- 술필수: 칵테일바, 와인바, LP바, 펍
- 양립: 퓨전포차, 고깃집, 이자카야, 호프, 곱창
- 일반: 한식·파스타·중식·분식 등

### 3.3 예산 집계

- 각자 `최소~최대` 범위 입력. 참여자 상한값 중 **낮은 쪽이 강하게** 작용(가장 빠듯한 사람 기준).
- 완전 최솟값은 위험 → **하위 25% 또는 최솟값~중앙값 사이**로 완충(후보 0개 방지). 부드러운 필터.

### 3.4 후보 0개 시 조건 완화 (F-11)

- 완화 순서: **예산 완충폭 확대 → 음식 카테고리 → 위치 반경** (술 제약은 끝까지 유지)
- 완화 시 사용자에게 **명시적 공지**: "조건에 맞는 곳이 없어 일부 조건을 완화해 추천했어요"

### 3.5 데이터 출처 / 웹서치 보완 (v1.4 신규)

- **카카오 플레이스 API는 평점·리뷰를 제공하지 않는다.** 따라서:
  - 식당 목록(이름·카테고리·좌표·거리)은 **카카오 API**로 수집.
  - 평점(`source_rating`)·분위기(`mood`)·가격대(`price_level`)는
    **Claude API 웹서치(다이닝코드·식신 우선)** 로 보완. **확인된 것만 저장(null 허용)**.
- **데이터 수집 방식**: 사전 키워드(이자카야·포차·고깃집·호프·곱창·한식·중식·일식·양식·분식·
  해산물·치킨·피자·스테이크·라멘 + FD6 전체)별로 검색 → `kakao_id` 기준 dedup →
  역당 150~250개 저장. 사용 키워드는 `search_keywords`(text[])에 누적 기록.
- **실시간 세션**: DB SQL 1차 필터 → Claude API 2차 최종 선별 (세션 중 카카오 API 호출 없음).
- **할루시네이션 차단**: 웹서치·선별 전 과정이 **카카오 수집 목록 기반**. 목록에 없는 식당 생성 금지.

## 4. 핵심 사용자 플로우

```
[생성자] 모임 생성(목적·최소인원·역·마감) → 초대 링크 공유
[참여자] 링크 입장 → 제약 질문 응답(술·예산·음식·분위기, 3스텝 이내)
[종료]   생성자 수동 종료 OR 마감시간 도달 (먼저 오는 것)
[시스템] 종료 시점 스냅샷 → 집계 → (웹서치 보완) → 후보 3~4곳 압축
[전원]   후보 👍 투표 → 최다 득표 자연 부상
```

- 참여자 기준 입장→응답→투표 **3스텝 이내**.
- 종료 전엔 집계하지 않음 → 입·퇴장 자유. 늦게 온 사람은 투표 단계부터 합류.
- **투표 = 추천된 후보 3~4곳 중에서 고르는 것**(취향 투표가 아님). 추천이 먼저, 투표가 나중.
- 👍는 앱인토스 user ID 기준 **개인별 1표, 중복 방지**.
- 최종 한 곳 선택은 **사용자 몫**(시스템은 최다 득표 하이라이트만, 확정하지 않음).

## 6. 데이터 / 외부 연동

- **식당 목록 데이터**: 카카오 플레이스 API (이름·카테고리·좌표·거리). **평점·리뷰는 미제공.**
- **평점·분위기·가격대 보완**: Claude API 웹서치(다이닝코드·식신). 확인된 것만 저장(null 허용).
  **API 키는 클라이언트에 박지 말고 서버/함수에 둔다.**
- **장소 타입 분류**: 사람이 다듬은 룩업 테이블(category_name 키워드 매핑). 코드로 적용, 재현 가능.
- **역 목록**: 백엔드 API로 받아온다. (서울 권역 + 인천 + 광명 기준, 권역→역 2단 선택)
- **백엔드/DB**: 프론트와 분리하지 않고 같은 레포에 서버리스(Supabase) 방식으로 구성한다.
  저장 대상: 그룹 / 참여자 / 제약 응답 / 후보 / 투표 / 식당캐시(search_keywords·source_rating 등).
  (스키마 확정은 백엔드 담당)

## 8. 작업 방식 (하네스)

1. 화면을 구현할 때는 `docs/screens/`의 해당 시안을 참고해 **TDS Mobile 컴포넌트로** 만든다.
2. 코드를 짠 뒤, **어떤 TDS 컴포넌트를 왜 썼는지 한 줄씩** 설명한다(사람 검수용).
3. 에러는 샌드박스에서 확인 → 메시지를 그대로 받아 수정 → 다시 실행(자가수정 루프).
4. 추천 엔진·예산·술 매핑·완화·웹서치 보완 로직은 3장의 규칙을 **정확히** 코드로 옮긴다. 임의 변경 금지.
